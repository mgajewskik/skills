# How-to Guides

Primary source: https://diataxis.fr/how-to-guides/

## What a how-to guide is

A how-to guide is a set of **practical directions** that helps an already-competent reader complete a real task or solve a real problem.

How-to guides are for readers who are **at work**.

## Defining rule

Write from the perspective of the **user’s goal**, not from the perspective of the tool’s features.

Good how-to guides answer to a human project.

Bad how-to guides merely describe what a tool can do.

## Core principles

- Focus on the real-world goal.
- Assume the reader has basic competence.
- Keep the guide practical: action and only action.
- Omit unnecessary teaching, theory, and reference detail.
- Describe a logical sequence.
- Allow branching, conditions, and judgment when reality requires them.
- Seek flow in both action and thinking.
- Title the guide so it says exactly what the reader can accomplish.

## Naming rule

Prefer titles like:
- `How to rotate signing keys`
- `How to deploy to staging with Docker`

Avoid vague or ambiguous titles like:
- `Signing keys`
- `Rotating signing keys`
- `Deployment`

## Language patterns

Prefer:
- `This guide shows you how to…`
- `If you want x, do y.`
- `In the case of z, use…`
- `Refer to the reference guide for the full option list.`

Avoid:
- tutorial voice that over-manages the learner
- conceptual essays
- exhaustive option catalogues that belong in reference

## How-to checklist

Pass when most are true:
- the task is meaningful outside the documentation itself
- the page assumes competence instead of trying to teach basics
- the steps stay focused on achieving the user’s goal
- conditional branches reflect real-world complexity where needed
- the page starts and ends in a practical, meaningful place
- extra context is linked out instead of embedded

Warning signs:
- the page is really a lesson for novices
- the page explains a feature instead of solving a user problem
- the steps are only tool motions with no human purpose
- the guide includes long reference tables or conceptual background

## Minimal rewrite moves toward how-to form

- rename the page around a user goal
- remove teaching and conceptual detours
- replace feature-centric structure with goal-centric sequence
- add branches only where real conditions change the right action
- move exhaustive options into reference and background into explanation
